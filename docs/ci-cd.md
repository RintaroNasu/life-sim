## 📌 概要

本プロジェクトでは、開発初期はシンプルなCIのみを導入し、MVP完成後にプロダクトの特性に応じてCD環境を設計・構築する方針とする。

---

## 🎯 目的

- 開発初期はスピードを優先し、過剰な構成を避ける
- MVP完成後に実運用を見据えたCI/CDパイプラインを構築する
- 継続的インテグレーションおよびデリバリの仕組みを段階的に整備する

---

## 🧩 現在の構成（CI）

### ブランチ戦略

- `main`：本番用ブランチ
- `feature/*`：機能開発ブランチ

### CI内容（GitHub Actions）

- Go
  - build

  - test

  - install
  - build
  - lint

### トリガー

- push（mainブランチ）
- pull_request

---

## 🚀 今後の構成（CD導入後）

### ブランチ戦略

- `main`：本番環境に対応
- `develop`：開発・検証用ブランチ
- `feature/*`：機能開発

### 環境

- staging環境（検証用）
- 本番環境

---

## 🚀 デプロイ方針

### staging環境

- `develop` ブランチにマージされたタイミングで自動デプロイを行う
- 開発中の機能の動作確認を目的とする

### 本番環境

- `main` ブランチへのマージをトリガーにデプロイを行う
- 本番反映は手動で実施する
- 意図しないデプロイを防ぐため、本番環境への反映は慎重に行う

## 🔁 デプロイフロー

```txt
feature → develop → staging環境
                ↓
              main → 本番環境
```

## 🧠 デプロイ戦略の意図

- staging環境は開発速度を優先し、自動デプロイとする
- 本番環境は安定性を優先し、手動でのリリースとする
- 環境ごとに役割を明確に分けることで、品質と速度のバランスを取る
